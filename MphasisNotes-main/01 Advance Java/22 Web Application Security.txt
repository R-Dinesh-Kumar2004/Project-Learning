
# Web Application Security (Java Web Apps)

---

## 1. Role of the Container in Web Application Security

### What is a Container?

> A **web container** (e.g. Tomcat) is responsible for running servlets, JSPs, filters, listeners — **and enforcing security rules**.

---

### What Security Responsibilities Does the Container Handle?

The container can:

* authenticate users
* authorize access to resources
* manage login/logout
* protect URLs
* enforce HTTPS
* manage security sessions

👉 **Developers declare security rules**
👉 **Container enforces them**

This separation is very important.

---

### Why container-based security is useful

Without container security:

* every servlet must check username/password
* duplicated logic
* error-prone

With container security:

* centralized
* standardized
* configurable via XML / annotations

---

## 2. Authentication vs Authorization (Must Clarify)

| Term           | Meaning                         |
| -------------- | ------------------------------- |
| Authentication | Who are you?                    |
| Authorization  | What are you allowed to access? |

Example:

* Login = authentication
* Access `/admin` = authorization

---

## 3. Four Authentication Models in Java Web Apps

Java EE / Jakarta EE defines **four standard authentication models**.

---

## 3.1. HTTP Basic Authentication

### How it works

* Browser shows default login popup
* Username & password sent in HTTP headers
* Base64 encoded (not encrypted)

---

### Configuration (conceptual)

Defined in `web.xml` with:

* security constraint
* login-config
* realm

---

### Pros

✅ Very easy
✅ No UI coding

---

### Cons

❌ Ugly browser popup
❌ Credentials sent every request
❌ Not secure without HTTPS

---

### Use Case

* internal tools
* testing
* quick admin protection

---

## 3.2. Form-Based Authentication

### How it works

* Custom HTML login form
* Container processes credentials
* User redirected after login

---

### Flow

```
User → login.jsp
      ↓
Container validates credentials
      ↓
Access granted / denied
```

---

### Pros

✅ Custom UI
✅ User-friendly
✅ Most commonly used

---

### Cons

❌ Slightly more configuration
❌ Still container-managed

---

### Use Case

* enterprise web applications
* portals
* dashboards

👉 This is the **most important model** for students.

---

## 3.3. Digest Authentication

### How it works

* Password is hashed before sending
* More secure than Basic auth
* Uses challenge–response

---

### Pros

✅ Better than Basic auth
✅ Password not sent in plain form

---

### Cons

❌ Still limited
❌ Rarely used today
❌ Not widely supported

---

### Use Case

* legacy systems
* constrained environments

---

## 3.4. Client-Certificate Authentication

### How it works

* Client uses SSL certificate
* Server validates certificate
* No username/password form

---

### Pros

✅ Very strong security
✅ No passwords

---

### Cons

❌ Complex setup
❌ Certificate management needed

---

### Use Case

* banking
* government portals
* B2B integrations

---

## 4. Comparison of Authentication Models

| Model       | UI            | Security | Complexity | Common Use |
| ----------- | ------------- | -------- | ---------- | ---------- |
| Basic       | Browser popup | Low      | Low        | Testing    |
| Form-based  | Custom        | Medium   | Medium     | Web apps   |
| Digest      | Browser popup | Medium   | Medium     | Rare       |
| Client cert | None          | High     | High       | Enterprise |

---

## 5. Forcing Encryption (HTTPS)

### Why encryption is required

Without encryption:

* passwords visible
* session hijacking
* data theft

---

### How container enforces encryption

Using **transport guarantee**:

```xml
<user-data-constraint>
   <transport-guarantee>CONFIDENTIAL</transport-guarantee>
</user-data-constraint>
```

This means:

> Application **must run over HTTPS only**.

---

### What happens internally

* If request comes via HTTP
* Container redirects to HTTPS
* Browser uses SSL/TLS

---

### Use Case

✔ login pages
✔ payment pages
✔ sensitive data

---

### One-line rule

> Authentication without HTTPS is useless.

---

## 6. Role of JAAS (Java Authentication and Authorization Service)

### What is JAAS?

> JAAS provides a **pluggable and extensible authentication framework**.

Instead of hardcoding:

* username/password logic
* database checks

JAAS allows:

* plug different authentication mechanisms
* without changing application code

---

### Why JAAS exists

* enterprises need:

  * LDAP
  * database
  * Active Directory
  * custom identity providers

JAAS abstracts authentication logic.

---

### How JAAS fits in web apps

```
Web App
   ↓
Container
   ↓
JAAS LoginModule
   ↓
Authentication Source
```

---

### Benefits of JAAS

✅ Pluggable authentication
✅ Enterprise-ready
✅ Separation of concerns
✅ Extensible

---

### Use Case

* corporate SSO
* enterprise authentication
* custom security providers

---

## 7. Big Picture: Who Does What?

| Responsibility        | Who handles it   |
| --------------------- | ---------------- |
| Authentication rules  | Developer        |
| Authorization rules   | Developer        |
| Credential validation | Container / JAAS |
| Session security      | Container        |
| HTTPS enforcement     | Container        |

---

## 8. Memory Rules (Very Important)

* Container enforces security
* Authentication ≠ Authorization
* Form-based auth is most common
* HTTPS is mandatory
* JAAS makes auth pluggable

---

## 9. What Students Should Remember Most

> **Security is declarative, not hardcoded.**

This prepares them for:

* Spring Security
* OAuth
* JWT
* SSO

---

## 🔟 One-line Summary

> Java web security is handled by the container using standard authentication models, HTTPS, and pluggable JAAS mechanisms.
