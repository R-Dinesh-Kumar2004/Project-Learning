
# Spring AOP

## 1. Introduction

**Spring AOP (Aspect Oriented Programming)** is a module of the Spring Framework.

It allows us to apply additional functionality across multiple classes and methods **without modifying their actual code**.

It follows the **Open/Closed Principle**:

* Open for extension
* Closed for modification

So our business logic stays clean, and common concerns are handled separately.

---

# 2. Why We Use Spring AOP (Common Use Cases)

Spring AOP mainly handles **cross-cutting concerns**, such as:

### 1. Logging & Auditing

* Log method calls, inputs, outputs
* Track user activities

### 2. Transaction Management

* Used internally by `@Transactional`
* Automatically manages commit & rollback

### 3. Security

* Check roles/permissions before method execution

### 4. Performance Monitoring

* Measure method execution time

### 5. Exception Handling

* Apply uniform exception handling
* Convert lower-layer exceptions into application exceptions

👉 In short: AOP handles functionality required in multiple places but not part of core business logic.

# 3. Important Terminology

1. Concern
2. Aspect
3. Advice
4. Pointcut
5. Join Point

---

# 4. Concern

A **Concern** is a functionality or responsibility of a system.

## Core Concern

The main business logic of the application.

Examples:

* `createRole()`
* `getUserById()`
* `existsByUsername()`

## Cross-Cutting Concern

Functionality required in many places but not part of business logic.

Examples:

* Logging
* Transaction handling
* Security checks

---

# 5. Aspect

An **Aspect** is a class that contains cross-cutting logic.

It defines **what** needs to be done, not **where**.

Example:

```java
@Aspect
@Component
public class LoggingAspect {
    // cross-cutting logic here
}
```

---

# 6. Advice

**Advice defines what action should run and when.**

It tells:

* When to execute (before/after/around)
* What to execute

## Types of Advice

1. **@Before** – Runs before method execution
2. **@AfterReturning** – Runs after successful execution
3. **@AfterThrowing** – Runs if method throws exception
4. **@After (Finally)** – Runs after method (success or failure)
5. **@Around** – Runs before and after, full control over execution

---

# 7. Join Point

A **Join Point** is a point in program execution where an aspect can be applied.

In Spring AOP:

* Usually method execution

Example:
Calling `makePayment()` is a join point.

---

# 8. Pointcut

A **Pointcut** defines where advice should run.

It is an expression that selects join points.

Example:

```java
execution(* spring.aop.services.PaymentServiceImpl.makePayment(..))
```

Meaning:

* Any return type (`*`)
* Method name: `makePayment`
* Any arguments (`(..)`)
* Inside specified class

---

## Common Pointcut Designators (PCD)

1. `execution` – Match method execution
2. `within` – Match methods inside specific class/package
3. `args` – Match methods with specific argument types
4. `this` – Match proxy object type
5. `target` – Match target object type
6. `@annotation` – Match annotated methods
7. `@within` – Match classes with annotation
8. `@args` – Match annotated arguments

Can be combined using:

* `&&`
* `||`
* `!`

---

# 9. Quick Concept Summary

| Term       | Meaning                                  |
| ---------- | ---------------------------------------- |
| Aspect     | The cross-cutting logic (WHAT)           |
| Join Point | Where aspect can apply                   |
| Advice     | Action taken (HOW & WHEN)                |
| Pointcut   | Expression selecting join points (WHERE) |

---

# 🔟 How Spring AOP Works

Spring AOP uses **proxy-based mechanism**.

* Spring creates a proxy object
* Intercepts method calls
* Executes advice at matched join points
* Then calls the actual method

We do not modify original methods.

---

# 1️1. Where to Place an Aspect?

An aspect:

* Must be a class
* Annotated with `@Aspect`
* Usually annotated with `@Component`

Example:

```java
@Aspect
@Component
public class LoggingAspect {

    @Before("execution(* com.example.service.*.*(..))")
    public void logBeforeMethod() {
        System.out.println("A method is about to be executed.");
    }
}
```

In this example:

* **Aspect** → `LoggingAspect`
* **Advice** → `logBeforeMethod()`
* **Pointcut** → `"execution(* com.example.service.*.*(..))"`
* **Designator** → `execution`
* **Join Points** → All matching service methods
* **Cross-cutting concern** → Logging

# 1️2. Development Setup Steps

1. Add Spring AOP dependency
* spring-aop 5.3.x
* aspectjweaver 1.9.x 

2. Enable auto proxy in WebConfig
* @EnableAspectJAutoProxy

3. Create a package for aspects (e.g., `app.orm.mvc.aspects`)

4. Create class with `@Aspect` and `@Component`

5. Define pointcuts and advices

# ✅ Single Aspect Class (Clean & Compact)

```java
@Aspect
@Component
public class BookDaoAspect {

    // 1. BEFORE – insertBook
    @Before("execution(* app.orm.mvc.dao.BookDaoImpl.insertBook(..))")
    public void beforeInsert(JoinPoint jp) {
        System.out.println("Before: " + jp.getSignature().getName());
    }
	
	
	
	
	
	
	
	
	
	
	
	
	
	
	
	
	

    // 2. AFTER (Finally) – modifyBook
    @After("execution(* app.orm.mvc.dao.BookDaoImpl.modifyBook(..))")
    public void afterModify(JoinPoint jp) {
        System.out.println("After (finally): " + jp.getSignature().getName());
    }


    // 3. AFTER RETURNING – getOneBook
    @AfterReturning(
        value = "execution(* app.orm.mvc.dao.BookDaoImpl.getOneBook(..))",
        returning = "result"
    )
    public void afterReturningGetOne(Object result) {
        System.out.println("AfterReturning: Result = " + result);
    }







    // 4. AFTER THROWING – deleteOneBook
    @AfterThrowing(
        value = "execution(* app.orm.mvc.dao.BookDaoImpl.deleteOneBook(..))",
        throwing = "ex"
    )
    public void afterThrowingDelete(Exception ex) {
        System.out.println("AfterThrowing: Exception = " + ex.getMessage());
    }





    // 5. AROUND – getAllBooks
    @Around("execution(* app.orm.mvc.dao.BookDaoImpl.getAllBooks(..))")
    public Object aroundGetAll(ProceedingJoinPoint pjp) throws Throwable {

        System.out.println("Around Before: getAllBooks");

        Object result = pjp.proceed();  // call actual method

        System.out.println("Around After: Returned list size = " + ((List<?>) result).size());

        return result;
    }
	










	@Around("execution(* app.orm.mvc.controller.*.*(..))")
    public Object logControllerMethods(ProceedingJoinPoint pjp) throws Throwable {

        // Get method details
        MethodSignature signature = (MethodSignature) pjp.getSignature();
        String methodName = signature.getName();
        String className = signature.getDeclaringType().getSimpleName();

        System.out.println("--------------------------------------------------");
        System.out.println("Class  : " + className);
        System.out.println("Method : " + methodName);

        // Print parameters
        Object[] args = pjp.getArgs();
        System.out.println("Arguments : " + Arrays.toString(args));

        // Execute actual method
        Object result = pjp.proceed();

        // Print return value
        System.out.println("Return Value : " + result);
        System.out.println("--------------------------------------------------");

        return result;
    }
	
	
}
```

---

# 🧠 What Each One Demonstrates

### 🔹 @Before → insertBook()

Runs before method execution

* Good for validation/logging

---

### 🔹 @After → modifyBook()

Runs after method finishes

* Executes whether success or exception

---

### 🔹 @AfterReturning → getOneBook()

Runs only if method completes successfully
Gives access to return value

---

### 🔹 @AfterThrowing → deleteOneBook()

Runs only if exception occurs
Used for error logging

(You can simulate by forcing a NullPointerException)

---

### 🔹 @Around → getAllBooks()

Most powerful advice

* Runs before
* Executes method manually
* Runs after

---

# 🔥 Summary

> BEFORE = only before
> AFTER = finally
> AFTER RETURNING = only on success
> AFTER THROWING = only on exception
> AROUND = full control
