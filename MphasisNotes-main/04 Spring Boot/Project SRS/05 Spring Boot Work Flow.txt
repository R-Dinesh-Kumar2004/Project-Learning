
# 📘 SbEcomApp – How Spring Boot Starts & Configures Everything

---

# 🧠 Big Picture Idea

When we start a Spring Boot application:

> Spring Boot automatically configures everything based on:

> * Classpath (dependencies)
> * application.properties
> * Annotations

We do not manually configure:

* Servlet
* DispatcherServlet
* EntityManager
* DataSource
* TransactionManager
* Hibernate Session
* MVC configuration

Spring Boot does all of that for us.

---

# 🧩 Memory Trick – 5 Major Phases

We can group startup into 5 phases:

1. Bootstrap Phase
2. Configuration Phase
3. Bean Creation Phase
4. JPA & Database Initialization Phase
5. Finalization Phase

---

# 🔹 Phase 1 – Bootstrap Phase

```
JVM Starts
↓
main() method executes
↓
SpringApplication.run()
```

At this point:

* Spring Boot is just starting
* No beans created yet
* No database connection yet

`SpringApplication.run()` is the real entry point into Spring Boot framework.

---

# 🔹 Phase 2 – Configuration Phase

```
Environment Prepared
(application.properties read)
↓
ApplicationContext Created
```

Here:

* Spring reads:

  * application.properties
  * environment variables
  * system properties
* Creates ApplicationContext container
* Prepares to register beans

ApplicationContext = Spring container.

---

# 🔹 Phase 3 – Bean Creation Phase

```
@ComponentScan runs
↓
Beans detected & registered
↓
@EnableAutoConfiguration runs
```

---

## 📌 What is @ComponentScan?

It scans the base package for:

```
@Component
@Controller
@RestController
@Service
@Repository
@Configuration
@ControllerAdvice
```

All of these are special forms of `@Component`.

So this part:

```
@Component
    |
@Controller  @Service  @Repository  @ControllerAdvice
    |
@RestController
```

Means:

All of these are detected as beans and registered in the container.

---

## 📌 What is @EnableAutoConfiguration?

This is the heart of Spring Boot.

It says:

> "Based on what is present in classpath, configure things automatically."

---

# 🔹 What Are Starters?

Starters are dependency bundles.

Example:

```
spring-boot-starter-web
spring-boot-starter-data-jpa
spring-boot-starter-validation
```

Each starter:

* Brings required libraries
* Brings auto-configuration classes
* Enables specific features

Example:

### If Web starter present:

Spring configures:

* DispatcherServlet
* MVC configuration
* JSON converters
* Embedded Tomcat

### If JPA starter present:

Spring configures:

* Hibernate
* EntityManagerFactory
* TransactionManager
* Repository infrastructure

Starters = feature switches.

---

# 🔹 How Auto Configuration Works Internally

```
Check classpath dependencies
```

If it finds:

* spring-web → Configure MVC
* spring-data-jpa → Configure JPA
* mysql driver → Configure DataSource

Spring Boot reads internal configuration classes inside:

```
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

Then activates matching configuration.

---

# 🔹 Phase 4 – JPA & Database Initialization

Now we enter database phase.

```
DataSource Created
↓
EntityManagerFactory Built
↓
Hibernate Initializes
↓
Entities Scanned
↓
Tables Created (ddl-auto)
↓
data.sql Executed
```

Let’s break this down.

---

## 1. DataSource Created

Spring reads:

```
spring.datasource.url
spring.datasource.username
spring.datasource.password
```

Then builds a DataSource bean.

This represents DB connection pool.

---

## 2. EntityManagerFactory Built

Because JPA starter is present, Spring:

* Creates EntityManagerFactory
* Connects it with DataSource

---

## 3. Hibernate Initializes

Hibernate:

* Scans @Entity classes
* Builds metadata
* Maps Java classes to DB tables

---

## 4. Tables Created

If:

```
spring.jpa.hibernate.ddl-auto=update
```

Hibernate automatically creates or updates tables.

---

## 5. data.sql Executed

If:

```
spring.sql.init.mode=always
```

SQL scripts run.

If:

```
spring.jpa.defer-datasource-initialization=true
```

SQL runs AFTER tables are created.

---

# 🔹 Phase 5 – Finalization Phase

```
ApplicationContext Fully Ready
↓
CommandLineRunner / ApplicationRunner Executes
↓
Embedded Tomcat Starts
↓
Application Ready
```

---

## CommandLineRunner

Runs AFTER:

* Beans created
* DB ready
* SQL executed

Used for:

* Initial data insertion
* Startup logic

---

## Embedded Server Starts

Because Web starter is present:

Spring Boot automatically:

* Configures embedded Tomcat
* Registers DispatcherServlet
* Maps controller endpoints

No web.xml required.

---

# 📊 Full Startup Flow (Grouped Version)

```
1. Bootstrap
JVM → main() → SpringApplication.run()

2. Configuration
Environment prepared → ApplicationContext created

3. Bean Detection
@ComponentScan → Beans registered

4. Auto Configuration
Check classpath → Configure Web/JPA/DataSource

5. Database Setup
DataSource → EntityManager → Hibernate → Tables → data.sql

6. Finalization
Context ready → Runner executes → Tomcat starts → App ready
```

---

# 🎯 What We Should Remember

Spring Boot works based on:

* Convention over configuration
* Classpath detection
* Auto-configuration
* Annotation scanning
* Embedded servers
* Smart defaults

We configure only what we need.

Spring configures the rest.

---

# 🧠 Big Concept

Spring Boot is not “magic”.

It:

* Reads dependencies
* Reads properties
* Applies conditional configuration
* Builds beans
* Starts server

All automatically.

---

# 🚀 Final One-Line Understanding

> Spring Boot analyzes our dependencies and properties, auto-configures required infrastructure (web, JPA, database, transactions), initializes beans, prepares database, and finally starts the embedded server to handle requests.
