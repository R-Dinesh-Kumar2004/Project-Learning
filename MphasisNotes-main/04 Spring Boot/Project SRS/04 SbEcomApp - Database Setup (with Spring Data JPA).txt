

# 📘 SbEcomApp – Database Setup (with Spring Data JPA)

---

# 1. Introduction to Spring Data JPA

Spring Data JPA is a powerful abstraction built on top of JPA (Java Persistence API).

It allows us to:

* Map Java objects to database tables
* Perform CRUD operations without writing SQL
* Automatically generate queries
* Manage relationships between tables
* Let Hibernate create database schema from entities

In a production backend system, we should never directly connect controllers to the database. 
We should use a structured persistence layer using Spring Data JPA.

---

# 2. What We Have to Set Up for JPA

To configure JPA properly in our project, we have to complete the following steps:

1. Add required dependencies
2. Configure database connection
3. Configure Hibernate properties
4. Create entity classes
5. Define relationships
6. Create repository interfaces
7. Configure SQL initialization (optional but useful)

---

# 3. Step 1 – Add Required Dependencies

We have to include:

* Spring Boot Starter Web
* Spring Boot Starter Data JPA
* MySQL Driver

Example dependencies:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>

<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

Spring Data JPA automatically includes Hibernate as the default JPA provider.

---

# 4. Step 2 – Configure Database Connection

In `application.properties`, we must define:

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/sb_ecom_app
spring.datasource.username=root
spring.datasource.password=1234

spring.jpa.database-platform=org.hibernate.dialect.MySQLDialect
```

These properties tell Spring:

* Which database to connect to
* What credentials to use
* Which SQL dialect to generate

---

# 5. Step 3 – Configure Hibernate Behavior

We must configure how Hibernate manages schema.

```properties
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
```

### Explanation:

### 🔹 ddl-auto=update

Hibernate should automatically create or update tables based on entity classes.

### 🔹 show-sql=true

We should enable this during development to see generated SQL in logs.

### 🔹 format_sql=true

Improves readability of SQL output.

---

# 6. Step 4 – Create Entity Classes

Entities represent database tables.

We have to use JPA annotations to map classes to tables.

---

## 📁 Category Entity

```java
@Entity
@Table(name = "categories")
public class Category {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;

    private String name;
    private String description;

    @OneToMany(mappedBy = "category", cascade = CascadeType.ALL)
    private List<Product> products;
}
```

### What We Should Understand:

* `@Entity` marks this as a JPA entity.
* `@Table` maps to a database table.
* `@Id` defines primary key.
* `@GeneratedValue` enables auto-increment.
* `@OneToMany` defines relationship.

---

## 📁 Product Entity

```java
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private BigDecimal price;
    private int stock;
    private String imagePath;

    @ManyToOne
    @JoinColumn(name = "category_id")
    private Category category;
}
```

### What We Should Understand:

* `@ManyToOne` defines foreign key relationship.
* `@JoinColumn` specifies column name in DB.

This creates:

```
products.category_id → categories.id
```

---

# 7. Step 5 – Create Repository Interfaces

We should create repository interfaces that extend `JpaRepository`.

Example:

```java
public interface CategoryRepository extends JpaRepository<Category, Integer> {
}
```

This automatically provides:

* save()
* findById()
* findAll()
* delete()
* count()

No implementation class is required.

Spring generates implementation at runtime.

---

# 8. Step 6 – Insert Initial Data

We may need initial data for testing.

There are two approaches:

### 1. SQL-based Initialization (`data.sql`)

```properties
spring.sql.init.mode=always
```

And create `data.sql`:

```sql
INSERT INTO categories (name, description)
VALUES ('Electronics', 'Electronic gadgets');
```

---

### 2. Java-based Initialization (`CommandLineRunner`)

```java
@Component
public class DataInitializer implements CommandLineRunner {

    @Autowired
    private CategoryRepository categoryRepository;

    @Override
    public void run(String... args) {

		Category fashion = new Category();
		fashion.setName("Fashion");
		fashion.setDescription("Clothing and accessories");

		categoryRepository.save(fashion);
    }
}
```

---

# 9. Important Issue – SQL Initialization Order

By default, Spring Boot executes `data.sql` before Hibernate creates tables.

Default startup order:

1. DataSource created
2. `data.sql` runs
3. Hibernate creates tables (by scanning entities)
4. Application continues

Problem:

If tables do not exist, SQL execution fails.

---

# 10. Correct Configuration for Proper Order

We must add:

```properties
spring.jpa.defer-datasource-initialization=true
```

This changes the order:

1. DataSource created
2. Hibernate creates tables
3. `data.sql` runs
4. CommandLineRunner runs
5. Application fully ready

This ensures SQL runs after schema creation.

---

# 11. Final Application Startup Flow

```
Application starts
↓
Hibernate scans @Entity classes
↓
Tables created (categories, products)
↓
data.sql executes (inserts data in table)
↓
CommandLineRunner executes (inserts more data)
↓
Application ready
```

While setting up Spring Data JPA, we should clearly understand:

* Entities define schema
* Hibernate generates tables
* Repositories abstract database access
* SQL initialization depends on schema creation
* Startup order matters
* Configuration properties control lifecycle

We must explicitly configure initialization order when needed.

# Transaction

---

## 🧠 Understanding What a Transaction Really Is

A transaction means:

* Either **all operations succeed**
* Or **all operations fail (rollback)**

It ensures:

✔ Atomicity
✔ Consistency
✔ Data integrity

---

# 📘 How Spring Manages Transactions

# 🔄 How Spring Manages Transactions

Spring manages transactions using **AOP (Proxy-based mechanism)**.

When we add:

```java
@Transactional
```

Spring does NOT modify our method directly.

Instead:

### 1. It creates a proxy around the service class.

When the method is called:

```
Controller
↓
Proxy (created by Spring)
↓
Actual Service Method
```

---

### 2. What the Proxy Does

Before method execution:

✔ Opens a database transaction

After method execution:

✔ If successful → COMMIT
✔ If RuntimeException occurs → ROLLBACK

---

### 3. Internally

Spring uses:

* `PlatformTransactionManager`
* Hibernate Session / EntityManager
* JDBC transaction underneath

---

# 🎯 In Simple Terms

`@Transactional` means:

> “Wrap this method inside a database transaction.”

Spring automatically:

* Starts transaction
* Executes method
* Commits or rolls back

---

# 🛒 In Our Order Example

When `placeOrder()` runs:

* Stock update
* Order insert
* OrderItem insert

All happen inside **one transaction**.

If anything fails → everything is undone.

---

# 🧠 Important Notes

* Only works on **public methods**
* Only works when called through Spring proxy
* Rolls back by default on **RuntimeException**

---

# 🚀 One-Line Summary

Spring manages transactions using AOP proxies that automatically begin, commit, or roll back database transactions around `@Transactional` methods.

---

Why `@Transactional` Is Used on Service Layer (Not Repository)

---

## 🎯 Short Answer

We use `@Transactional` on the **Service layer** because:

> A transaction represents a **business operation**, not a single database operation.

And business logic lives in the **Service layer**.

---

# 🚫 Why NOT on Repository?

Repositories usually perform **single database operations**, like:

* save()
* findById()
* delete()

But real business operations involve **multiple repository calls**.

If we put `@Transactional` on repository methods:

* Each method would run in its own transaction.
* We would lose control over the full business process.

That is not correct design.

---

# ✅ Why On Service Layer?

Service layer contains:

* Business logic
* Multiple repository calls
* Validation
* Stock update
* Order calculation

So the entire business process must be wrapped inside ONE transaction.

That’s why:

```
@Service
@Transactional
public class OrderServiceImpl {
```

---

# 🛒 Example: Order Management in SbEcomApp

Let’s analyze what happens when a user places an order.

---

## 🧾 Order Placement Process

When user places an order:

1. Fetch User
2. Fetch Product
3. Check stock
4. Reduce stock
5. Create Order
6. Create OrderItems
7. Save Order
8. Calculate total
9. Save everything

That is **not one DB operation**.
It is a complete business workflow.

---

# 💥 What If There Is No Transaction?

Imagine this scenario:

* Stock was reduced
* Order saved
* But saving OrderItems fails

Now we have:

❌ Stock reduced
❌ Order partially saved
❌ Database inconsistent

That is a serious production bug.

---

# ✅ What Happens With `@Transactional`

If we annotate:

```java
@Transactional
public Order placeOrder(OrderRequestDTO request)
```

Spring does:

1. Opens transaction
2. Executes all DB operations
3. If everything succeeds → commit
4. If any exception occurs → rollback everything

So if OrderItem save fails:

✔ Stock reduction is rolled back
✔ Order insert is rolled back
✔ Database returns to original state

---

# 🔍 Technical Working Behind The Scenes

When `@Transactional` is applied:

Spring:

* Creates a proxy around the service
* Starts transaction before method execution
* Commits transaction if method completes successfully
* Rolls back if RuntimeException occurs

---

# 📦 How It Works in Our App

In `OrderServiceImpl`:

```java
@Service
public class OrderServiceImpl implements OrderService {

    @Transactional
    public Order placeOrder(OrderRequestDTO request) {

        User user = userRepository.findById(request.getUserId()).orElseThrow();

        Product product = productRepository.findById(request.getProductId()).orElseThrow();

        if (product.getStock() < request.getQuantity()) {
            throw new BusinessException("Insufficient stock");
        }

        product.setStock(product.getStock() - request.getQuantity());
        productRepository.save(product);

        Order order = new Order();
        order.setUser(user);

        orderRepository.save(order);

        return order;
    }
}
```

If any line throws exception:

➡ Entire transaction rolls back.

---

# 🎯 Why Not Annotate Repository?

If repository methods had transactions individually:

* save(product) would commit immediately
* save(order) would commit separately

Then rollback across multiple operations becomes impossible.

Transaction must wrap the whole workflow.

---

# 🧠 Rule of Thumb

| Layer      | Should Have @Transactional? |
| ---------- | --------------------------- |
| Controller | ❌ No                        |
| Service    | ✅ Yes                       |
| Repository | ❌ No (managed by service)   |

---

# 🔥 Key Architectural Principle

> Transactions represent business boundaries, not database boundaries.

Business boundary = Service layer.

---

# 🎓 Important points

> "Repository handles database operations.
> Service handles business operations.
> Transactions belong to business operations."

That’s the clean conceptual clarity.

---

# 🚀 Final Summary

We use `@Transactional` on Service because:

✔ It wraps multiple repository calls
✔ It ensures atomic business operations
✔ It prevents partial updates
✔ It maintains data consistency
✔ It is industry best practice
